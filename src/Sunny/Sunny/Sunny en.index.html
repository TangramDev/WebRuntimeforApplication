Common Layout Engine

Common Layout Engine is a Multi-Level, Gridded, Nestable, easy to describe UI layout description engine.

       The so-called grid is a window matrix with m rows and n columns (a grid with 1 row and n columns can be understood as a tab window with n tabs), and nestable means that each "grid" in the matrix can be Multiple grid layers are dynamically nested, and only one layer is visible at runtime.

       Generally speaking, a layout may be uncertain at design time and can be dynamically generated at runtime. Due to different scenarios, permissions and other factors, different dynamic matching may be carried out. In fact, flexible layout is the result of dynamic changes of multi-layer grid.

       There are two ways to describe grid: one is based on XML, which can be used in various programming languages; the other is based on DOM element of web page. The advantage of this scheme is that it is easy to connect with mainstream Internet technology.

Grid

Grid is a window matrix with M rows and N columns. As a container, a specific "grid" in the grid can host a new grid or a programmable object, such as .Net UserControl, WinForm or WPF object, or other objects implemented by C + +, such as MFC CView……

In a specific web page or an XML document, a DOM element corresponding to "grid" can be defined in the following format. Here, a description of DOM corresponding to a grid structure of 3 rows and 3 columns is given. Please refer to Sunny.index.html

<content_grid target="mainClient"> <cluster> <xobj rows="3" cols="3" width="150,100,100" height="200,180,100"> <xobj objid="SunnyCtrl.UserControl1,SunnyCtrl"></xobj> <xobj objid="SunnyCtrl.UserControl2,SunnyCtrl"></xobj> <xobj objid="SunnyCtrl.UserControl3,SunnyCtrl"></xobj> <xobj objid="SunnyCtrl.UserControl3,SunnyCtrl"></xobj> <xobj objid="nucleus"></xobj> <xobj objid="SunnyCtrl.UserControl1,SunnyCtrl"></xobj> <xobj objid="SunnyCtrl.UserControl1,SunnyCtrl"></xobj> <xobj objid="SunnyCtrl.UserControl3,SunnyCtrl"></xobj> <xobj objid="SunnyCtrl.UserControl2,SunnyCtrl"></xobj> </xobj> </cluster> </content_grid>

A grid without child nodes can host specific programmable objects, or the runtime can host a new grid structure, depending on the specific programming requirements. When we clearly need a grid to hold a specific object, we need to specify the "objid" attribute of the object.

Currently Sunny can support several types of programmable objects, including:
  1. .NET UI Object
  2. ActiveX Control
  3. C++ objects conforming to certain specifications
  4. Web Page

For .Net UI, we generally refer to WinForm, UserControl and WPF, and UWP support will be added in subsequent versions. When a grid loads the. Net UI, the objid attribute is the AssemblyQualifiedName of the object. Usually, it is a string such as "A.B, C", where a is the object's namespace, B is the class name, and C is the component name.

For example, suppose we have a .Net component library xxx.dll , where there is a formx object whose namespace is "mynamespace", then objid is“ MyNamespace.FormX,XXX ", and the corresponding grid nodes are as follows：

<xobj objid="MyNamespace.FormX,xxx"></xobj>
Tab

If a grid element contains child nodes, and the grid does not have "rows" and "cols" attributes, sunny interprets it as a "tab", and the number of tabs it contains is the number of its child nodes.

Common Web Runtime

Common Web Runtime(CWR) is the dynamic link library version of Google Chromium Project

       The CWR contains all the code for the chromium project. Its goal is to make the windows desktop application implement the built-in modern browser model according to the consistent scheme, and make the host desktop software become the modern web browser naturally. Different from the traditional browser, the application integrated with CWR has its own application object model. Because all functions of browser are provided based on DLL mode, CWR does not know what the host application is and what object model the host application contains. CWR solves the connection problem between host model and browser standard model, the real browser model can only be determined after a specific host application is defined.

       The DOM model of the host application will be a superset of the DOM model of the standard browser. In other words, the host application system will provide parts of the application model that cannot be provided in the standard browser. The actual DOM model of the host application may include members that are not part of the standard browser DOM model, which means that the host application supports both standard and application based web pages (there is no application based web page in standard browser mode). Because the host application has all the functions of chromium project built in, application developers can develop new application content for host application based on Web technology, and make host application become software ecosystem based on Web technology.

       In terms of DOM, CWR adds the support of. Net framework, which allows DOM elements for. Net UI to be written with DOM rules on Web pages. That is to say, WinForm, UserControl, WPF and other objects can be embodied in the actual development as part of page elements, which is very important for the formation of application ecology.

Common Eclipse Runtime

The Common Eclipse Runtime(CER) is the dynamic link library version of the Eclipse Project

       It contains all the code for the eclipse project. Its goal is to make the windows desktop application implement the built-in eclipse model according to the consistent scheme, and make the host desktop software become the startup program of eclipse naturally.

       Different from the traditional eclipse, the application integrated with CER (Common Eclipse Runtime) has its own application object model. Since all the functions of the eclipse part are provided based on the dll mode, CER does not know in advance whether the host application is What, what object model the host application contains, CER solves the connection problem between the host model and the eclipse standard model, so the real eclipse model can only be determined after the host application is clear. At the same time, the OSGi model of the host application will be larger than the standard eclipse OSGi model. In other words, the host application system will provide application model parts that are not provided in the standard eclipse.

       The actual OSGi model of the host application may include members that are not part of the standard eclipse OSGi model, which means that the host application supports both standard eclipse components and application-based eclipse components, and does not provide specific application-oriented components in the standard Eclipse mode eclipse components. Because the host application has all the functions of the Eclipse project built-in, application developers can develop new application content for the host application based on Eclipse technology, so that the host application becomes a software ecosystem based on Java technology.
